# gstack 使用指导文档

本文档面向“先计划后执行”的开发方式，目标是把 `gstack` 的常见模式完整覆盖，并明确每种场景下该怎么选。

核心原则只有一句：

**先用合适的模式把问题定义清楚，再进入实现、验证、发布。**

---

## 1. gstack 的定位

`gstack` 不是单一的“写代码助手”，而是一套覆盖完整交付流程的工作流工具。它把开发拆成几个阶段：

- 需求澄清
- 方案设计
- 实现开发
- 调试排障
- 代码审查
- 浏览器验证
- 发布收尾
- 安全控制
- 复盘沉淀

如果你平时是“小步快跑、边做边改”，它会显得偏重。  
如果你是“先计划、再执行、最后验证和发版”，它会非常顺手。

---

## 2. 模式总览

| 模式 | 主要用途 | 最适合的场景 | 风格特点 |
|---|---|---|---|
| `/office-hours` | 澄清需求和目标 | 需求模糊、方向不清、想先讨论值不值得做 | 先问问题，挑战前提，偏产品思维 |
| `/plan-ceo-review` | 产品层面审视方案 | 想把目标做对，或判断该不该扩大/缩小范围 | 偏创始人视角，强调价值和取舍 |
| `/plan-eng-review` | 技术方案审查 | 要落地实现前先把架构定稳 | 偏工程负责人视角，关注边界、状态、数据流 |
| `/plan-design-review` | 设计方案审查 | UI/交互/视觉改动 | 偏设计评审，关注体验完整性和一致性 |
| `/design-consultation` | 构建设计系统 | 从零做新界面、新品牌、新视觉体系 | 更像设计顾问，不只是审图 |
| `/investigate` | 根因分析 | Bug、故障、行为异常 | 先查原因，反对盲修 |
| `/review` | 代码审查 | 功能完成后检查风险 | 偏结构性问题、隐性 bug、生产风险 |
| `/qa` | 测试并修复 | 功能完成后做验证 | 会测试，也会修正问题 |
| `/qa-only` | 只测试不修复 | 想先拿问题报告 | 纯报告输出，不改代码 |
| `/browse` | 浏览器真实交互验证 | 登录、表单、截图、响应式、页面交互 | 是“执行器”，不是“分析器” |
| `/setup-browser-cookies` | 导入真实浏览器登录态 | 测试需要登录的页面 | 让 headless 浏览器直接进入已登录状态 |
| `/ship` | 发版交付 | 准备合并、打包、发版 | 偏发布工程，处理版本、变更、PR |
| `/document-release` | 同步文档 | 代码发完之后更新文档 | 避免文档和代码脱节 |
| `/retro` | 周期复盘 | 周报、迭代总结、团队回顾 | 看趋势、看健康度、看产出 |
| `/cso` | 安全审计 | 安全敏感模块、权限、依赖、泄露风险 | 偏安全负责人视角 |
| `/careful` | 高风险命令保护 | 删除、强制推送、重置、数据库危险操作 | 软拦截，允许覆盖 |
| `/freeze` | 限定编辑范围 | 调试时防止误改其他目录 | 控制修改边界 |
| `/guard` | 安全模式合集 | 高风险环境、生产排障 | 结合 `careful` 和 `freeze` |
| `/unfreeze` | 解除编辑限制 | 之前冻结后需要扩大范围 | 恢复全局编辑 |
| `/codex` | 第二意见 | 想要独立审查、对抗式挑战 | 适合和 review 互补 |
| `/gstack-upgrade` | 升级 gstack | 需要更新工具本身 | 工具维护用途 |

## 3. 各模式详细说明

### 3.1 `/office-hours`

**用处**

- 把模糊需求变成清晰需求。
- 确认“你到底想解决什么问题”。
- 防止一开始就做偏。

**特点**

- 会主动追问背景、目标、约束、用户、现状。
- 更像产品讨论，而不是技术实现。
- 常用于需求刚出现、信息不完整、方向未定的时候。

**适用场景**

- 你只有一句话需求，但不确定要做到什么程度。
- 你怀疑这个需求本身并不值得做。
- 你想先把问题边界问清楚。

**不适合的场景**

- 需求已经很明确，只差实现。
- 只是一个明确 bug 修复。
- 你已经有完整方案，不需要再脑暴。

---

### 3.2 `/plan-ceo-review`

**用处**

- 从产品价值角度审视方案。
- 判断范围该扩大还是收缩。
- 帮你决定“做什么最值”。

**特点**

- 偏 founder / CEO 视角。
- 会追问用户价值、差异化、长期影响。
- 常常会提出“是不是应该做得更大一点”或者“这个功能可以更小更锋利”。

**适用场景**

- 你在做产品方向判断。
- 你想把一个小需求升级成更有价值的方案。
- 你对范围拿不准，想先定策略。

**不适合的场景**

- 你只想快速实现一个小改动。
- 你已经完全确定产品目标，不需要再改范围。

---

### 3.3 `/plan-eng-review`

**用处**

- 在写代码前把技术方案定稳。
- 审查架构、数据流、状态、边界、错误处理和测试策略。

**特点**

- 偏技术负责人或工程经理视角。
- 会关心：
  - 数据怎么流
  - 谁拥有状态
  - 边界条件在哪里
  - 失败怎么恢复
  - 需要哪些测试覆盖
- 适合把“能不能做”变成“怎么做最稳”。

**适用场景**

- 小功能改动，但你想先把实现方式定下来。
- 多模块联动，怕改坏别的地方。
- 后端、状态管理、权限、持久化相关改动。

**不适合的场景**

- 纯文案修改。
- 很小的 CSS 微调。
- 已经写完，只需要验收。

---

### 3.4 `/plan-design-review`

**用处**

- 在实现前审查 UI/UX 方案。
- 发现页面层面的缺口，比如状态不全、交互断裂、响应式不完整。

**特点**

- 偏设计师视角。
- 会检查：
  - 空状态
  - loading 状态
  - error 状态
  - 移动端表现
  - 视觉层级
  - 文案是否完整
- 会明确指出设计方案里容易忽略的问题。

**适用场景**

- 前端页面改版。
- 新增弹窗、面板、表单、列表等 UI。
- 需要在实现前把视觉和交互锁定。

**不适合的场景**

- 纯后端逻辑改动。
- 没有界面变化的功能。

---

### 3.5 `/design-consultation`

**用处**

- 从零建立设计系统。
- 适合新产品、新品牌、新大页面体系。

**特点**

- 不只是“审查”，而是“构建”。
- 会从字体、颜色、布局、节奏、动效、品牌气质出发。
- 通常会输出可复用的设计原则和预览内容。

**适用场景**

- 你要启动一个新项目。
- 你发现当前 UI 没有统一风格。
- 你想建立系统性的视觉规范。

**不适合的场景**

- 只是给某个按钮改颜色。
- 单个小功能迭代，不需要重做设计系统。

---

### 3.6 `/investigate`

**用处**

- 诊断 bug 或异常的根本原因。
- 把“现象”变成“原因”。

**特点**

- 强调先调查，再修改。
- 不接受“试试看”的盲修方式。
- 会追代码路径、日志、输入输出、条件分支、边界数据。

**适用场景**

- 页面行为不对。
- 接口结果异常。
- 某个功能偶发失败。
- 你不知道问题是前端、后端还是状态同步导致的。

**不适合的场景**

- 需求本身是正常的新功能，不是 bug。
- 你已经知道原因，只需要修复。

---

### 3.7 `/review`

**用处**

- 对已经完成的改动做代码审查。
- 发现结构性问题、隐性风险和未来维护成本。

**特点**

- 偏严格 review。
- 不只是看代码能不能跑，还看：
  - 有没有漏掉边界条件
  - 会不会引入竞态
  - 是否破坏信任边界
  - 是否有 SQL / 权限 / 依赖 / 状态污染问题
- 更像“上线前的第二双眼睛”。

**适用场景**

- 功能已经写完。
- 准备合并代码。
- 想提前发现生产风险。

**不适合的场景**

- 还没开始写代码。
- 你只是要查 bug 根因，那应该先用 `/investigate`。

---

### 3.8 `/qa`

**用处**

- 真正验证功能是否按预期工作。
- 发现问题后顺手修复，再验证。

**特点**

- 会结合代码变化和页面实际状态做检查。
- 是“测试 + 修复 + 再验证”的闭环。
- 更适合功能完成后的最后阶段。

**适用场景**

- 一个功能已经开发完。
- 你想确认前端交互、路由、表单、状态更新是否正常。
- 你准备把它交给别人用。

**不适合的场景**

- 还没实现完。
- 你只是做概念讨论。

---

### 3.9 `/qa-only`

**用处**

- 只做测试，不做修复。
- 适合先要一份客观问题清单。

**特点**

- 输出偏报告。
- 不会主动改代码。
- 适合做验收前的质量盘点。

**适用场景**

- 你想先看看系统里有什么问题。
- 你要把问题交给别人修。
- 你不希望测试过程中自动改动代码。

**不适合的场景**

- 你希望直接帮你修掉问题。
- 你想要完整的测试修复闭环，那应该用 `/qa`。

---

### 3.10 `/browse`

**用处**

- 用真实浏览器操作页面。
- 验证真实交互、视觉呈现和响应式效果。

**特点**

- 适合“看得见摸得着”的验证任务。
- 能点按钮、填表单、看弹窗、截屏、调尺寸。
- 比纯代码分析更接近最终用户体验。

**适用场景**

- 登录流程。
- 多步骤表单。
- 列表筛选、弹窗、拖拽等交互。
- 响应式布局检查。
- 截图比对。

**不适合的场景**

- 纯后端逻辑。
- 不需要 UI 验证的改动。

---

### 3.11 `/setup-browser-cookies`

**用处**

- 把你本地真实浏览器的登录态导入到测试浏览器。

**特点**

- 适合有权限、登录态、私有后台的页面。
- 减少重复登录。
- 让浏览器 QA 更接近真实使用环境。

**适用场景**

- 测试管理后台。
- 测试需要登录的页面。
- 反复进行多次浏览器验证。

**不适合的场景**

- 公共页面无需登录。
- 你不需要真实身份态。

---

### 3.12 `/ship`

**用处**

- 把一个已完成改动推向正式交付流程。

**特点**

- 关注的是“交付完成”而不是“代码写完”。
- 常见动作包括：
  - 合并或同步主分支
  - 跑测试
  - 处理版本号
  - 更新 changelog
  - 创建或更新 PR
- 偏发布工程流程。

**适用场景**

- 功能已验证完成。
- 你准备发 PR 或发版。
- 想把最后一公里流程交给系统处理。

**不适合的场景**

- 你还在方案阶段。
- 功能还没稳定。

---

### 3.13 `/document-release`

**用处**

- 让文档跟随代码变化更新。

**特点**

- 会检查 README、架构说明、贡献指南、待办事项等。
- 目标是减少“代码改了但文档过时”的问题。
- 更像发版后的收尾工作。

**适用场景**

- 你刚发完版本。
- 新增了用户可见功能。
- 改了配置、权限、流程、命令等需要写进文档的内容。

**不适合的场景**

- 还没发布。
- 改动很小，不影响文档。

---

### 3.14 `/retro`

**用处**

- 对过去一段时间的交付进行复盘。
- 看产出趋势、效率、质量、风险。

**特点**

- 更关注模式和趋势，而不是单次任务。
- 会分析提交、节奏、测试质量、团队分工等。

**适用场景**

- 周会。
- 迭代总结。
- 研发流程优化。

**不适合的场景**

- 单个小功能开发当下。

---

### 3.15 `/cso`

**用处**

- 安全审计。
- 发现潜在安全漏洞和供应链风险。

**特点**

- 偏安全负责人视角。
- 会关注：
  - 权限控制
  - 敏感信息暴露
  - 依赖风险
  - 注入类问题
  - 数据泄露
  - AI / LLM 的信任边界

**适用场景**

- 涉及认证、授权、密钥、用户数据。
- 即将上线高风险功能。
- 你想做一次全面安全体检。

**不适合的场景**

- 普通 UI 文案修改。
- 低风险内部逻辑改动。

---

### 3.16 `/careful`

**用处**

- 防止危险命令误操作。

**特点**

- 会对高风险命令给出明显提醒。
- 重点防止：
  - `rm -rf`
  - `git reset --hard`
  - 强制推送
  - 危险数据库操作
- 允许用户明确覆盖，但默认会拦一下。

**适用场景**

- 需要删文件、重置分支、处理生产环境命令。
- 你知道自己在做什么，但想多一层保护。

**不适合的场景**

- 日常无风险开发。

---

### 3.17 `/freeze`

**用处**

- 限定本轮只能改某个目录。

**特点**

- 适合把修复范围固定住，避免误伤别的模块。
- 对调试特别有价值。
- 能减少“顺手改了不该改的文件”。

**适用场景**

- 你只想改 `src/components` 或某个特定模块。
- 你在做局部 bugfix。
- 你担心范围失控。

**不适合的场景**

- 需要大范围重构。
- 需要跨目录联动修改。

---

### 3.18 `/guard`

**用处**

- 把安全限制和编辑限制合成一个更强的保护模式。

**特点**

- 同时包含：
  - 危险命令保护
  - 目录编辑限制
- 适合高风险排障和生产环境相关工作。

**适用场景**

- 生产问题处理。
- 高风险发布前操作。
- 你想最大程度避免误操作。

**不适合的场景**

- 普通日常开发，通常太重。

---

### 3.19 `/unfreeze`

**用处**

- 解除之前的编辑范围限制。

**特点**

- 从局部受限恢复为全局可编辑。
- 常用于从局部修复切换到更大范围重构。

**适用场景**

- 你刚才限制过范围，现在确认需要扩大范围。
- 任务已经进入下一阶段。

---

### 3.20 `/codex`

**用处**

- 让另一个 Codex 再给你一层独立意见。

**特点**

- 可以做：
  - 代码 review
  - 对抗式挑战
  - 带上下文的咨询
- 适合发现你自己和主 agent 容易忽略的问题。

**适用场景**

- 你想要第二意见。
- 你对一个方案没把握。
- 你怀疑主方案有盲点。

**不适合的场景**

- 非常简单的改动，没必要增加一层审查成本。

---

### 3.21 `/gstack-upgrade`

**用处**

- 升级 gstack 工具本身。

**特点**

- 识别安装方式。
- 更新到最新版本。
- 属于工具维护，不是项目功能开发。

**适用场景**

- 你想用最新技能行为。
- 文档或功能提示你需要升级。

## 4. 场景化推荐流程

下面给出最常见场景的推荐组合。

### 4.1 需求很模糊

推荐流程：

`/office-hours -> /plan-ceo-review -> /plan-eng-review -> implement -> /review -> /qa`

说明：

- 先把目标问清楚。
- 再确认值不值得做，做多大。
- 然后锁实现方案。
- 最后验证和收尾。

---

### 4.2 需求明确的小功能改动

推荐流程：

`/plan-eng-review -> implement -> /review -> /qa`

说明：

- 这是日常最实用的最小闭环。
- 适合大多数小需求。

---

### 4.3 前端 UI 改动

推荐流程：

`/plan-eng-review -> /plan-design-review -> implement -> /browse -> /qa -> /review`

说明：

- 技术方案和视觉方案都先定好。
- 用浏览器真实验证交互。
- 最后做代码结构审查。

---

### 4.4 新增一个页面或复杂交互

推荐流程：

`/plan-eng-review -> /plan-design-review -> implement -> /browse -> /qa -> /review -> /ship`

说明：

- 页面类需求通常需要更完整的规划。
- 浏览器验证基本不能省。

---

### 4.5 后端逻辑或状态管理改动

推荐流程：

`/plan-eng-review -> implement -> /review -> /qa`

说明：

- 重点在边界、状态、错误处理和测试。
- 设计审查通常可省略。

---

### 4.6 线上 bug 修复

推荐流程：

`/investigate -> /plan-eng-review -> implement -> /review -> /qa`

说明：

- 先根因分析。
- 再设计最小修复方案。
- 最后验证没有引入新问题。

---

### 4.7 难复现 bug

推荐流程：

`/investigate -> /browse -> /qa-only -> /plan-eng-review -> implement -> /qa`

说明：

- 浏览器操作和纯报告都可能帮助复现。
- 先确认现象，再修复。

---

### 4.8 安全敏感改动

推荐流程：

`/cso -> /guard -> implement -> /review -> /qa -> /ship`

说明：

- 涉及权限、密钥、敏感数据时不要省略安全检查。
- `/guard` 可以降低误操作风险。

---

### 4.9 准备发版

推荐流程：

`/review -> /qa -> /ship -> /document-release`

说明：

- 先审查、再验证、再发版、最后更新文档。
- 这是最完整的交付收口。

---

### 4.10 需要登录态的后台验证

推荐流程：

`/setup-browser-cookies -> /browse -> /qa`

说明：

- 先导入登录态。
- 再在浏览器里验证真实操作路径。

---

### 4.11 你想要第二意见

推荐流程：

`/plan-eng-review -> /codex -> implement -> /review`

说明：

- 先做主方案。
- 再让独立视角挑战一下。
- 适合复杂改动。

---

## 5. 日常开发的最小推荐套路

如果你想把 gstack 用在日常迭代里，不想太重，可以直接记这几条：

### 5.1 普通小功能

`/plan-eng-review -> implement -> /review -> /qa`

### 5.2 前端小功能

`/plan-eng-review -> /plan-design-review -> implement -> /browse -> /qa`

### 5.3 线上 bug

`/investigate -> implement -> /review -> /qa`

### 5.4 交付前

`/review -> /qa -> /ship`

### 5.5 有安全风险

`/cso -> /guard -> implement -> /review -> /qa`

---

## 6. 什么时候不要用重模式

以下情况通常不需要走完整流程：

- 只改一个文案
- 只调一个样式
- 只修一行明显的逻辑错误
- 明确知道原因且修复范围很小
- 不涉及页面、不涉及发布、不涉及权限、不涉及状态流

这种时候，直接用最小闭环即可。

---

## 7. 推荐的使用原则

### 7.1 先计划后执行

先用 `plan-*` 把方案想清楚，再写代码。

### 7.2 先调查后修复

如果是 bug，先 `investigate`，不要先拍脑袋改。

### 7.3 改完一定验证

功能类变更默认要走 `review` 和 `qa`。

### 7.4 UI 改动要浏览器验证

只看代码不够，至少一次 `/browse`。

### 7.5 高风险场景要加保护

`/careful`、`/freeze`、`/guard` 不是装饰，是防事故。

## 8. 一页速查版

- 需求不清楚：`/office-hours`
- 要先判断值不值得做：`/plan-ceo-review`
- 要先锁技术方案：`/plan-eng-review`
- 要先审 UI：`/plan-design-review`
- 要从零做设计系统：`/design-consultation`
- 遇到 bug 先查原因：`/investigate`
- 写完后查结构问题：`/review`
- 写完后做功能验证：`/qa`
- 只想看问题不想改：`/qa-only`
- 要真实浏览器操作：`/browse`
- 要导入登录态：`/setup-browser-cookies`
- 要发版：`/ship`
- 要同步文档：`/document-release`
- 要做复盘：`/retro`
- 要做安全审计：`/cso`
- 要防危险命令：`/careful`
- 要限制编辑范围：`/freeze`
- 要双重保护：`/guard`
- 要解除冻结：`/unfreeze`
- 要第二意见：`/codex`
- 要升级工具：`/gstack-upgrade`

---

## 9. 结论

如果你是“先计划后执行”的工作方式，`gstack` 最适合的日常套路是：

`/plan-eng-review -> implement -> /review -> /qa`

当需求不清楚时，加 `/office-hours`。  
当是 UI 改动时，加 `/plan-design-review` 和 `/browse`。  
当是 bug 时，先用 `/investigate`。  
当涉及安全和高风险操作时，用 `/cso`、`/careful`、`/freeze` 或 `/guard`。

这套方式的目标不是让流程变重，而是让每一步都更少返工。
